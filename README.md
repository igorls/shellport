Share your shell over the network with **end-to-end AES-256-GCM encryption**, **TOTP two-factor authentication**, a **custom canvas terminal emulator**, and **multi-session tabs** — all packed into one file powered by [Bun](https://bun.sh). No xterm.js. No node_modules. No config files.

```bash
shellport server
```

Then open the URL printed in your terminal. That's it.

## Why ShellPort?

Most web terminal tools either force you to haul in xterm.js + a Node.js runtime, or sacrifice encryption entirely. ShellPort takes a different approach:

| Problem | ShellPort's Answer |
|---|---|
| Heavy frontend deps (xterm.js, hterm) | **NanoTermV2** — custom Canvas2D renderer, ~50 KB, zero deps |
| No built-in encryption | **E2E AES-256-GCM** with PBKDF2 key derivation (100K iterations) |
| No 2FA support | **TOTP 2FA** — pair with any authenticator app via QR code |
| Complex setup | **Single command** — `bunx shellport server` |
| Multiple services needed | **One binary** — HTTP server, WebSocket, PTY, and frontend all included |
| Can't share publicly | **Tailscale integration** — one flag for public HTTPS (requires Tailscale CLI) |

## Features

## Security

ShellPort is designed with security-first principles:

### Encryption & Authentication

- **Per-session cryptographic salt** — Each session derives a unique salt from server and client nonces, preventing precomputation attacks
- **AES-256-GCM** encryption with 12-byte random IV per message
- **PBKDF2 key derivation** (100,000 iterations, SHA-256)
- **128-bit session secrets** auto-generated by default
- Secret passed via URL fragment (`#secret`) — **never sent to the server**
- **Proof-of-knowledge authentication** — Client must successfully decrypt a server nonce to prove key possession

### TOTP Two-Factor Authentication

- **RFC 6238 TOTP** — Compatible with Google Authenticator, Authy, 1Password, etc.
- **QR code pairing** — Scan directly from your terminal (zero-dependency QR generator)
- **Persistent secret** — Pair once, the TOTP secret is saved locally and reused across restarts
- **Works with or without encryption** — TOTP can be the sole auth factor or layered on top of E2E encryption
- Use `--totp-reset` to regenerate the pairing secret

### Security Modes

| Mode | Command | Auth | Encryption | Use Case |
|------|---------|------|------------|----------|
| **Full 2FA** | `shellport server --totp` | Secret + TOTP | AES-256-GCM | Maximum security |
| **Encrypted** | `shellport server` | URL secret | AES-256-GCM | Quick secure sessions |
| **TOTP only** | `shellport server --no-secret --totp` | TOTP code | None* | Remote via VPN/Tailscale |
| **Plaintext** | `shellport server --no-secret` | None | None | Localhost dev only |

\* When using TOTP-only mode over Tailscale/WireGuard, the VPN tunnel provides encryption.

### Access Control

- **Interactive approval mode** (default ON) — Server host must approve each connection from their terminal
- **Rate limiting** — Max 5 connection attempts per IP per minute
- **Session limits** — Maximum 10 concurrent PTY sessions
- **Strict origin validation** — WebSocket only accepts connections from same origin (localhost bypass requires `--dev` flag)

### Transport Security

- **Security headers** — CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy
- **WebSocket subprotocol** — Protocol versioning enforced
- **Input validation** — All control messages validated with bounds checking
- **Buffer limits** — OSC/DCS sequences capped at 64KB to prevent memory exhaustion

### Terminal Security

- **Clipboard write confirmation** — OSC 52 clipboard operations require user approval (configurable)
- **Safe environment inheritance** — Only whitelisted environment variables passed to PTY
- **No telemetry** — Zero external network calls

### Command Line Security Flags

```bash
# Full security — encrypted + TOTP 2FA
shellport server --totp

# Encrypted only (default)
shellport server

# TOTP only — ideal for VPN/Tailscale
shellport server --no-secret --totp

# Development mode (relaxes origin checks)
shellport server --dev

# Reset TOTP pairing
shellport server --totp-reset
```

### Known Limitations

- No forward secrecy — if the secret is compromised, all traffic can be decrypted
- Connection approval requires terminal access to the server
- TOTP secret is stored in `~/.shellport-totp-secret` — protect this file

> ⚠️ **For production use**, enable TOTP (`--totp`) and keep approval mode on. Use `--no-secret` only with TOTP over a VPN, or on isolated trusted networks.

### 🖥️ NanoTermV2 — Canvas Terminal Emulator
- **Canvas2D** hardware-accelerated rendering (no DOM nodes)
- VT100 / VT220 / xterm escape sequence parsing
- 256-color + **truecolor** (24-bit) support
- Alternate screen buffer (vim, htop, tmux)
- Text selection with clipboard integration
- Mouse tracking (X10, Normal, SGR modes)
- Bracketed paste mode
- UTF-8 streaming decoder
- **~50 KB** — usable as a standalone library

### 📡 Server & Connectivity
- Native **Bun PTY** API — true terminal, zero-latency
- WebSocket binary framing with sequenced message ordering
- Multi-session support with **tmux-style sidebar tabs**
- Terminal resize forwarding
- **Tailscale** integration — `serve` or `funnel` in one flag
- CLI client for terminal-to-terminal connections

### 📦 Zero Friction
- **Zero runtime dependencies** — only `@types/bun` for development
- Cross-compiled single binaries for Linux, macOS, and Windows
- ~2,400 lines of TypeScript + JavaScript total

## Install

### Quick Start (requires Bun)

```bash
bunx shellport server
```

### Prebuilt Binary

Download from [GitHub Releases](https://github.com/igorls/shellport/releases):

```bash
curl -fsSL https://github.com/igorls/shellport/releases/latest/download/shellport-linux-x64 -o shellport
chmod +x shellport
./shellport server
```

### From Source

```bash
git clone https://github.com/igorls/shellport.git
cd shellport
bun install
bun run dev          # Run locally
bun run build        # Build single binary (current platform)
bun run build:binaries  # Cross-compile all platforms
```

## Usage

### Server

```bash
# Start with auto-generated secret (recommended)
shellport server
# → Prints: 🌐 Open in browser: http://localhost:7681/#<random-secret>

# Enable TOTP 2FA (scans QR on first run)
shellport server --totp

# TOTP only — no encryption (use over VPN/Tailscale)
shellport server --no-secret --totp

# Fixed secret (not recommended — prefer auto-generated)
shellport server --secret your-secret-here

# Plaintext mode (trusted network only)
shellport server --no-secret

# Custom port
shellport server --port 8080

# Public via Tailscale Funnel
shellport server --tailscale funnel
```

The server prints the full URL with the secret — just open it in your browser.

### CLI Client

Connect from another machine's terminal:

```bash
shellport client ws://host:7681/ws --secret your-secret-here
```

### Options

| Option | Description | Default |
|---|---|---|
| `--port, -p` | Server port | `7681` |
| `--secret, -s` | Fixed encryption secret | *(auto-generated)* |
| `--no-secret` | Disable encryption (plaintext) | *(off)* |
| `--totp` | Enable TOTP 2FA (shows QR on first run) | *(off)* |
| `--totp-reset` | Regenerate TOTP pairing secret | *(off)* |
| `--no-approval` | Disable connection approval prompts | *(off)* |
| `--allow-localhost, --dev` | Allow localhost origin bypass | *(off)* |
| `--tailscale` | `serve` or `funnel` | *(disabled)* |
| `--quiet, -q` | Suppress non-essential output | *(off)* |

#### Environment Variables

| Variable | Description |
|---|---|
| `SHELLPORT_SECRET` | E2E encryption key (avoids exposing secret in `ps`) |
| `SHELLPORT_APPROVAL_MODE` | Set to `disabled` to skip approval prompts |

## NanoTermV2 — Standalone Library

The canvas terminal emulator can be used independently in any web project:

```html
<script src="https://unpkg.com/shellport/src/frontend/nanoterm.js"></script>
<script>
  const term = new NanoTermV2(document.getElementById('terminal'), data => {
    ws.send(data); // Handle user input
  });

  term.write('Hello, world!\r\n');
</script>
```

Or via module import:

```js
import "shellport/nanoterm";

const term = new NanoTermV2(container, sendFn, {
  fontSize: 14,
  fontFamily: "'JetBrains Mono', monospace",
  cursorStyle: 'block',    // 'block' | 'underline' | 'bar'
  cursorBlink: true,
  scrollback: 10000,
  theme: {
    background: '#0a0a0a',
    foreground: '#e0e0e0',
    cursor: '#a78bfa',
  }
});
```

### When to pick what

- **ShellPort** — You want encrypted terminal sharing with the smallest footprint and zero dependencies. Great for DevOps, SSH-less remote access, and embedding a terminal in your own app via NanoTermV2.
- **ttyd** — You need a battle-tested, high-performance web terminal with file transfer support. Best for kiosk/dashboard terminals.
- **sshx** — You need real-time collaboration with multiple people (live cursors, chat). Best for pair programming and live demos.
- **tmate** — You need instant tmux-style session sharing with SSH clients. Best for pair debugging when both parties use a terminal.
- **Upterm** — You need SSH-based session sharing that works behind NATs. Best for CI/CD debugging and remote pairing.
- **Wetty** — You need a web frontend to an existing SSH server. Best for web-based SSH access to existing infrastructure.

## Architecture

```
src/
├── index.ts          # CLI entry — argument parsing & routing
├── server.ts         # HTTP + WebSocket + PTY server (Bun.serve)
├── client.ts         # CLI WebSocket client (raw terminal mode)
├── crypto.ts         # AES-256-GCM engine (PBKDF2 key derivation)
├── totp.ts           # TOTP 2FA (RFC 6238) — secret, QR, verify
├── qr.ts             # QR code generator (terminal-rendered)
├── types.ts          # TypeScript types, constants, SeqQueue
└── frontend/
    ├── index.html    # HTML shell
    ├── styles.css    # Terminal + sidebar UI styles
    ├── nanoterm.js   # Canvas terminal emulator (standalone library)
    ├── app.js        # Session manager & UI logic
    └── build.ts      # HTML assembler (inlines all assets)
```

### Data Flow

```
Browser                          Server
  │                                │
  │  1. Server sends nonce         │
  │◀──────────────────────────────│
  │                                │
  │  2. Client sends nonce         │
  │──────────────────────────────▶│
  │  (per-session key derived)     │
  │                                │
  │  3. TOTP challenge (if enabled)│
  │◀──────────────────────────────│
  │                                │
  │  4. TOTP code response         │
  │──────────────────────────────▶│──▶ Verify code
  │                                │
  │  WebSocket (binary frames)     │
  │──────────────────────────────▶│
  │  [iv(12)][AES-GCM ciphertext] │
  │                                │──▶ PTY (shell process)
  │◀──────────────────────────────│
  │  [iv(12)][AES-GCM ciphertext] │
  │                                │
  └── URL fragment #secret         │
      (never leaves the browser)   │
```

## Security Model

| Layer | Mechanism |
|---|---|
| **Default Mode** | Random 128-bit secret auto-generated per session |
| **Per-Session Salt** | SHA-256(server_nonce \|\| client_nonce \|\| prefix) |
| **Key Derivation** | PBKDF2 · 100,000 iterations · SHA-256 |
| **Encryption** | AES-256-GCM (authenticated encryption) |
| **IV** | 12-byte random nonce per message |
| **Authentication** | Mutual nonce exchange + successful decryption |
| **TOTP 2FA** | RFC 6238 · SHA-1 · 6 digits · 30s window |
| **Connection Approval** | Interactive terminal prompt (default ON) |
| **Secret Transport** | URL fragment (`#secret`) — not sent in HTTP requests |
| **Origin Validation** | Strict same-origin enforcement |
| **Rate Limiting** | 5 attempts per IP per minute |
| **Session Cap** | Max 10 concurrent PTY sessions |
| **Security Headers** | CSP, X-Frame-Options, nosniff |
| **TOTP-only Mode** | `--no-secret --totp` for VPN/Tailscale setups |
| **Plaintext Mode** | Available with `--no-secret` (for localhost / VPN) |

> ⚠️ ShellPort is designed for **private terminal access**, not multi-tenant hosting. Keep approval mode enabled for production use.
