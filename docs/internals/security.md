# Security Model

ShellPort is designed with security-first principles, implementing multiple layers of protection for encrypted terminal sessions. This page documents the complete security architecture.

## Encryption Layers

### Per-Session Cryptographic Salt

Each session derives a unique cryptographic salt that prevents precomputation attacks:

```
salt = SHA-256(server_nonce || client_nonce || secret_prefix)
```

- **Server Nonce** — Random 32-byte nonce generated by the server
- **Client Nonce** — Random 32-byte nonce generated by the client
- **Secret Prefix** — Derived from the URL fragment secret

This ensures that even if an attacker knows the secret, they cannot precompute session keys.

### Key Derivation (PBKDF2)

The session encryption key is derived using PBKDF2 with high iteration count:

| Parameter | Value |
|-----------|-------|
| Algorithm | PBKDF2-SHA-256 |
| Iterations | 100,000 |
| Salt | Per-session unique (from nonces) |
| Output Length | 256 bits (32 bytes) |

```typescript
const key = await crypto.subtle.deriveKey(
  { name: 'PBKDF2', salt, iterations: 100000, hash: 'SHA-256' },
  baseKey,
  { name: 'AES-GCM', length: 256 },
  false,
  ['encrypt', 'decrypt']
);
```

### AES-256-GCM Encryption

All terminal data is encrypted using AES-256-GCM, which provides both confidentiality and authenticity:

| Property | Value |
|----------|-------|
| Cipher | AES-256 |
| Mode | GCM (Galois/Counter Mode) |
| IV Size | 12 bytes (96 bits) |
| Tag Size | 16 bytes (128 bits) |
| Key Size | 32 bytes (256 bits) |

Each message is encrypted with a fresh random IV:

```
[12-byte IV][AES-GCM ciphertext][16-byte auth tag]
```

### IV (Initialization Vector)

- **Size** — 12 bytes (96 bits) per message
- **Generation** — Cryptographically secure random bytes
- **Uniqueness** — Never reused within a session
- **Purpose** — Provides semantic security; same plaintext encrypts to different ciphertext

## Authentication Mechanisms

### Proof-of-Knowledge Protocol

ShellPort uses a mutual nonce exchange for authentication:

1. **Server generates** a random challenge nonce
2. **Server encrypts** its own nonce with the derived session key
3. **Client decrypts** and returns the nonce
4. **Server verifies** the returned nonce matches

This proves the client possesses the correct session key without transmitting it.

### TOTP Two-Factor Authentication

ShellPort implements RFC 6238 Time-based One-Time Passwords:

| Parameter | Value |
|-----------|-------|
| Algorithm | HMAC-SHA-1 |
| Digits | 6 |
| Period | 30 seconds |
| Window | ±1 (90 second total) |

#### Setup Flow

1. Server generates a random 160-bit secret
2. Secret is encoded as a base32 string
3. Server renders a QR code in the terminal for scanning
4. Client stores the secret for future verification

#### QR Code Format

The QR code contains an `otpauth://` URI:

```
otpauth://totp/ShellPort:user@host?secret=JBSWY3DPEHPK3PXP&issuer=ShellPort
```

#### Verification

- User enters 6-digit code from their authenticator app
- Server computes expected TOTP using stored secret
- Code is valid if it matches within ±1 period (clock drift tolerance)

#### Supported Authenticators

- Google Authenticator
- Authy
- 1Password
- Any RFC 6238 compliant TOTP app

### Nonce Exchange Sequence

```
Browser                          Server
  │                                │
  │  1. Server sends nonce         │
  │◀──────────────────────────────│
  │                                │
  │  2. Client sends nonce         │
  │──────────────────────────────▶│
  │  (per-session key derived)     │
  │                                │
  │  3. TOTP challenge (if enabled)│
  │◀──────────────────────────────│
  │                                │
  │  4. TOTP code response         │
  │──────────────────────────────▶│──▶ Verify code
  │                                │
```

## Transport Security

### URL Fragment Secret Transport

The encryption secret travels in the URL fragment (`#secret`):

```
http://localhost:7681/#K7x9Pq2mNvLs8Yt3W
```

- **Never sent to server** — URL fragment is client-side only
- **Browser History** — May be stored in browser history
- **Referer Header** — Not included in HTTP requests
- **Server Logs** — Not visible in server access logs

### WebSocket Security

- **Subprotocol Versioning** — Enforced protocol version in WebSocket handshake
- **Origin Validation** — Strict same-origin enforcement
- **Localhost Bypass** — Requires `--dev` flag to relax origin checks

### Security Headers

The HTTP responses include security headers:

| Header | Value | Purpose |
|--------|-------|---------|
| Content-Security-Policy | strict | Prevent XSS injections |
| X-Frame-Options | SAMEORIGIN | Prevent clickjacking |
| X-Content-Type-Options | nosniff | Prevent MIME sniffing |
| Referrer-Policy | strict-origin | Control referrer information |

### Input Validation

- **Bounds Checking** — All control messages validated before processing
- **Buffer Limits** — OSC/DCS sequences capped at 64KB
- **Type Validation** — Message types checked before dispatch

## Terminal Security Features

### Clipboard Operations (OSC 52)

Clipboard read/write operations require user confirmation:

- **Default: Ask** — User prompted to approve each clipboard operation
- **Configurable** — Can be disabled via server configuration
- **Timeout** — Operations time out after 10 seconds

### Environment Variable Filtering

Only whitelisted environment variables are passed to the PTY:

```typescript
const ALLOWED_ENV = [
  'TERM', 'COLORTERM', 'LANG', 'LC_*',
  'SHELL', 'HOME', 'USER', 'PATH'
];
```

Blocked variables include:
- `SHELLPORT_*` internal variables
- `WS_*` WebSocket variables
- `SECRET`, `PASSWORD`, `TOKEN` patterns

### Connection Approval Mode

Interactive approval requires manual confirmation:

- **Default: ON** — Server operator must approve connections
- **Timeout** — 30 second timeout for approval prompt
- **Rejection** — Connection dropped if denied or timeout
- **Override** — Available via `--no-approval` flag

### Rate Limiting

Brute-force protection against authentication attempts:

| Limit | Window | Scope |
|-------|-------|-------|
| 5 attempts | 1 minute | Per IP address |

### Session Limits

| Limit | Reason |
|-------|--------|
| 10 concurrent sessions | Resource management |
| Per-session unique key | Isolation |

### No Telemetry

- **Zero external network calls** — All functionality is local
- **No analytics** — No tracking or telemetry
- **No CDN dependencies** — Everything bundled inline

## Security Modes

ShellPort supports four security configurations:

| Mode | Command | Encryption | TOTP | Approval |
|------|---------|------------|------|----------|
| **Full Security** | `shellport server` | AES-256-GCM | Required | Required |
| **Encrypted Only** | `shellport server --no-totp` | AES-256-GCM | Disabled | Required |
| **TOTP Only** | `shellport server --no-secret` | None | Required | Required |
| **Plaintext** | `shellport server --no-secret --no-totp` | None | Disabled | Required |

### Mode Recommendations

| Use Case | Recommended Mode |
|----------|------------------|
| Default / Public internet | Full Security |
| Trusted internal network | Encrypted Only |
| VPN/Tailscale connection | TOTP Only |
| Local development only | Plaintext |

## Known Limitations

::: warning
Understanding these limitations is important for secure deployment.
:::

### No Forward Secrecy

- If the session secret is compromised after the session ends, all historical traffic can be decrypted
- Use over VPN/Tailscale for additional protection
- Consider rotating secrets regularly

### TOTP Secret Storage

- The TOTP pairing secret is stored in `~/.shellport-totp-secret`
- This file should be protected (chmod 600)
- Loss of this file requires `--totp-reset` to re-pair

### Connection Approval Requires Terminal Access

- Someone with terminal access to the server must approve connections
- Not suitable for fully remote/unattended deployments without VPN

## Security Checklist

For production deployments:

- [ ] Keep TOTP enabled (default)
- [ ] Keep approval mode enabled (default)
- [ ] Use `--no-secret` only over VPN/Tailscale
- [ ] Protect `~/.shellport-totp-secret` file
- [ ] Use `--tailscale serve` or `--tailscale funnel` for public HTTPS
- [ ] Review firewall rules (default port: 7681)
- [ ] Consider IP allowlisting for sensitive deployments
